# 月次コーチングセッション プロンプト集

用途: personal-dashboardアプリの外側で行う、トレーニング・食事・KPIの設計セッション。
新しいClaudeチャットに貼り付けて使う。成果物は週間プログラムJSON(REQUIREMENTS.md §2.3a の契約形式)。

---

## 初回セッション用プロンプト(KPI目標設定 + 初回プログラム設計)

```
筋肥大目的のトレーニングコーチとして、KPI目標の設定と4週間のトレーニングプログラム設計を手伝ってほしい。

# 長期目標
なかやまきんに君・横川尚隆のような筋肉質な体になり、それを維持すること。

# 私の現状
- トレーニング歴と現在のログ: [主要種目の現在の重量×回数を部位ごとに書く。例: ベンチプレス60kg×8回。2回目以降はアプリのコーチング用エクスポート(G-F11)を貼る]
- 体組成: 身長[  ]cm / 体重[  ]kg / 体脂肪率[  ]%(わかれば)
- 周囲径(わかれば): 胸[ ] 腕[ ] ウエスト[ ] 大腿[ ]
- 環境: エニタイムフィットネス(マシン・フリーウェイト・ダンベル利用可)
- 使える頻度: 週[  ]回、1回あたり[  ]分
- 睡眠の現状: [①ダッシュボードの睡眠時間の平均などを貼る]
- 食事の制約: 自炊は手間を最小限にしたい。コンビニ・冷凍・プロテイン活用は可

# 方法論の参照
メニュー設計にあたっては、山本義徳氏(101理論、高強度・低ボリューム志向)と、なかやまきんに君(基本重視・フォームと継続重視)の公開している方法論を検索して参照し、私の現状レベルにどう適用するかを明示すること。両者の考えが食い違う点は、初中級者向けにどちらを採るか理由付きで判断すること。

# 求める成果物
1. KPI目標値の提案: 結果指標(体重・体脂肪率・周囲径の3ヶ月/6ヶ月目標)とプロセス指標(主要種目のe1RM目標、週あたり部位別セット数、実施率、タンパク質g/日、睡眠時間)。現状値からの実現可能性に言及すること
2. 4週間の週間プログラム: 週[  ]回の分割法(どの曜日にどの部位か)、各日の種目・セット数・レップ範囲。種目数は1日4種目以内でシンプルに
3. 食事方針: タンパク質目標g/日と、手間最小の具体例(朝昼晩+間食の定型パターンを2〜3案)。カロリー計算が必要なら概算でよい
4. 上記2を、最後に以下のJSON形式で出力すること(私のアプリに取り込むため):

{
  "schemaVersion": 1,
  "programName": "YYYY-MM 筋肥大期 v1",
  "validFrom": "YYYY-MM-DD",
  "validUntil": "YYYY-MM-DD",
  "weeklySchedule": [
    {"dayOfWeek": "Mon", "focus": "部位", "exercises": [
      {"name": "種目名", "sets": 3, "reps": "8-10", "note": "重量目安など"}
    ]}
  ],
  "nutritionTargets": {"proteinGramsPerDay": 0, "note": ""},
  "bodyMetricsToTrack": ["weight", "bodyFatPct", "chest", "arm", "waist", "thigh"]
}

# 進め方
- 不足している情報があれば、設計の前に私に質問すること
- 提案は「確定的な根拠があるもの」と「個人差が大きく試行が必要なもの」を分けて示すこと
```

## 2回目以降(月次レビュー)用プロンプト

```
先月設計してもらった筋肥大プログラムの月次レビューと、来月のプログラム改訂をしたい。

# 先月のプログラム
[前回のプログラムJSONを貼る]

# 実績データ
[アプリのコーチング用エクスポート(G-F11)を貼る: e1RM変化、部位別ボリューム、実施率、ボディ記録、タンパク質達成率、睡眠要約]

# 主観メモ
- うまくいったこと: [ ]
- きつかった・できなかったこと: [ ]
- 関節の違和感など: [ ]

# 求めること
1. KPIに対する進捗評価(順調/停滞の判定と、停滞ならボトルネックの仮説)
2. 来月のプログラム改訂(重量・ボリューム・種目の変更は最小限で。変更には理由を付けること)
3. 改訂版を前回と同じJSON形式で出力
4. 山本義徳・なかやまきんに君の方法論と照らして、今の進め方に修正すべき点があれば指摘

# 進め方
- データから言えることと、データ不足で判断できないことを分けること
- 実施率が低い場合は、プログラムを増やすのではなく減らす方向をまず検討すること
```

## 運用メモ

- 頻度: 月1回(プログラムのvalidUntilに合わせる)。停滞や違和感があれば臨時開催
- 初回セッションで決めたKPI目標値は `docs/` に保存し、毎回のレビューで参照する
- JSONを受け取ったら、アプリのプログラム取込(G-F6)→ ics生成(G-F7)→ iPhoneカレンダー登録、までがセッション後のルーティン
- 食事の具体メニュー化(買い物リスト等)が必要な場合は、同セッション内で続けて依頼してよい(アプリには入れない)
