# personal-dashboard 要件定義書

作成日: 2026-07-04
対象: Claude Code による実装
スコープ: ①バイタルダッシュボード / ③ジムログ可視化 / ④習慣化 の3機能を単一PWAとして開発する。
(②週末プランナーは本リポジトリのスコープ外。Claude.ai Artifactとして別途実装する)

## 0. プロジェクト全体方針

### 0.1 確定済みの前提(変更不可)

| 項目 | 決定内容 | 理由 |
|---|---|---|
| プラットフォーム | PWA(iPhone + ホーム画面追加) | ネイティブiOSアプリは不採用(Xcode未経験、署名運用コスト) |
| 対象ブラウザ | ユーザーの既定ブラウザ(現在はChrome想定)。**Safari固有機能に依存しない** | Web Push不使用のためブラウザ非依存が可能。IndexedDBはブラウザ単位で分離されるため**単一ブラウザ運用を前提**とする |
| 設計原則 | **自動化ファースト**(§0.5) | 反復的な手動操作は本プロジェクトの最重要削減対象 |
| ホスティング | GitHub Pages(静的サイト) | 無料枠のみで運用する制約 |
| サーバー | なし。完全クライアントサイド | 健康データを外部送信しない(プライバシー要件) |
| データ永続化 | ブラウザ内 IndexedDB | 毎回のファイル再読込を不要にする |
| データ入力 | ファイルインポート(iOSファイルピッカー経由) | iOSショートカット自作エクスポート(日次) + ヘルスケア標準エクスポートXML(初回バックフィル) |
| 言語 | UIは日本語 | 利用者は開発者本人のみ |

### 0.2 推奨技術スタック(Claude Code側で合理的な代替提案は可)

- React + TypeScript + Vite
- グラフ: Recharts または ECharts(エラーバー描画が必要な点に注意 → §3)
- IndexedDB ラッパー: Dexie.js
- PWA: vite-plugin-pwa(manifest + service worker、オフライン閲覧対応)
- デプロイ: GitHub Actions → GitHub Pages

### 0.4 実装順序(セッション分割の指示)

1. **フェーズ1**: PWAシェル + 共通層(core) + ①バイタル → 最初のデプロイ
2. **フェーズ2a**: ③記録+可視化(G-F1〜G-F5: セット記録・種目マスタ・中央値グラフ・e1RM)
3. **フェーズ2b**: ③プログラム実行支援(G-F6〜G-F11: プログラム取込・ics通知・ボディ記録・KPIサマリー)
4. **フェーズ3以降**: ④習慣化(docs/habit-algorithm.md のフェーズ表に従い複数セッションに分割)。ジム通い自体は④に `physical` 活動として登録し、継続支援は④が担う(③は通知の中身=メニューを担当し、役割が重複しないこと)

各フェーズは独立したClaude Codeセッションで実施する。フェーズをまたぐ共通知識はCLAUDE.mdに記載する。

### 0.5 設計原則: 自動化ファースト(全機能に優先適用)

ユーザーは**手動操作の多さがアプリの利用継続を阻害する最大要因**と認識しており、可能な限りの自動化を強く望んでいる。すべての設計判断において以下を適用すること。

1. **反復操作は設計課題として扱う**: 毎日・毎週発生する手動操作は、実装前に「自動化・統合・削減できないか」を必ず検討する。削減不能な場合のみ、その理由とともに残す
2. **タップ数を設計指標にする**: 日次運用に必要なユーザー操作のタップ数を機能設計時に明示し、最小化する。日次の定常運用は「朝の通知1タップ+アプリ内数タップ」を超えないこと
3. **データ連携はファイル選択より自動連携を優先**: ファイルピッカーは常にフォールバックとし、URL連携(V-F7)・オートメーション・購読方式を第一候補とする
4. **開いたら終わっている状態を作る**: ユーザーがアプリを開いた時点でデータ取込・更新が完了している動線(ショートカット→URL起動)を標準とし、「取込のために開く」設計を避ける
5. **設定は一度きり**: 初回セットアップ(ショートカット作成、カレンダー購読等)に手間がかかるのは許容するが、定常運用に手作業を残さないこととのトレードオフで判断する

## 1. 機能① バイタルダッシュボード

### 1.1 目的

Apple Watch + Soxai Ring(いずれもApple Health同期済み)の睡眠・バイタルデータを日単位の時系列で俯瞰し、回復状態を把握する。

### 1.2 データフロー(2系統)

**注: Health Auto Exportは有料機能制限があるため不採用。以下の無料構成とする。**

```
[日次更新 = Plan A]
Apple Health
  → iOSショートカット(「ヘルスケアサンプルを検索」。オートメーションで毎朝7時トリガー+実行前確認ON → 通知1タップで実行)
  → JSONをbase64化し、PWAのURL(#import=...)を「URLを開く」で起動
  → PWAが起動時に自動取込(V-F7) → そのまま④チェックイン・③タンパク質記録へ続く単一動線

[初回バックフィル = Plan B]
ヘルスケアapp「すべてのヘルスケアデータを書き出す」(標準機能・無料)
  → export.zip(export.xml)
  → 本PWAのXMLインポート(ファイルピッカー) → 対象5指標のみ抽出 → IndexedDBへ保存
```

- URLフラグメントはサーバーに送信されず(そもそも静的ホスティング)、取込後に即破棄する(V-F7)
- URL長の安全域を考慮し、ショートカット側は heart_rate を日次要約(その日の最小/最大/平均)に集約してから出力する。詳細分解能が必要になった場合のみファイル経由(V-F3)を使う

### 1.3a 日次エクスポートJSONスキーマ(v2。ショートカット側とPWA側の契約)

設計方針: **ショートカット側は単純に、集計ロジックはPWA側に寄せる**(ショートカットの保守性・作成難易度を優先)。睡眠のみレコード単位(正午区切り帰属・ソース選択をアプリ側で行うため)、他は日次集計値。

```json
{
  "schemaVersion": 2,
  "exportedAt": "2026-07-05T07:00:12+09:00",
  "date": "2026-07-04",
  "sleep": [
    {"start": "2026-07-04T23:12:00+09:00", "end": "2026-07-05T06:40:00+09:00", "stage": "Asleep", "source": "SOXAI Ring"}
  ],
  "steps": 8421,
  "hrvSdnn": 45.2,
  "restingHr": 52,
  "heartRate": {"min": 47, "max": 142, "avg": 68}
}
```

- `date` は集計対象日(前日)。steps/hrvSdnn/restingHr/heartRate はこの日の集計値として保存する
- `sleep` はレコード配列。`stage` はショートカットのValue出力文字列をそのまま格納(例: "In Bed", "Asleep", "Core", "Deep", "REM", "Awake"。実出力に合わせてパーサ側でマッピング)
- 数値がクォート付き文字列で出力される場合がある(ショートカットの仕様揺れ)。**パーサは数値/文字列の両方を受理**すること
- ショートカットの実出力が本スキーマと食い違う場合は、**実出力を正としてスキーマを改訂**し、本書を更新すること

### 1.3 機能要件

| ID | 要件 | 詳細 |
|---|---|---|
| V-F1 | 表示指標 | 合計睡眠時間 / HRV(SDNN) / 安静時心拍 / 歩数 / 心拍数(日次要約: 最小・最大・平均) |
| V-F2 | 時間軸 | 最小粒度は1日。表示範囲: 週 / 月 / 3ヶ月 / 年 の切替 |
| V-F3 | インポート | (a) §1.3a 定義JSONの日次取込、(b) ヘルスケア標準export.xmlのバックフィル取込。いずれも**同一期間・同一指標・同一ソースは上書き(冪等)**。複数ファイルの追加取込に対応 |
| V-F3b | XML処理 | export.xmlは数百MBになりうるため、ストリーミングパースで対象5指標のみ抽出し、メモリに全展開しない |
| V-F4 | 永続化 | インポート済みデータはIndexedDBに保持。アプリ再訪時は即描画 |
| V-F5 | 重ね合わせ | 2指標の同時表示(例: 睡眠時間×HRV)。第2軸を使用 |
| V-F6 | データ管理 | 保存済みデータの期間・件数の確認と全削除機能 |
| V-F7 | URLフラグメント自動取込 | `#import=<base64(JSON)>` 付きでPWAが開かれた場合、§1.3a契約JSONとして自動パース→IndexedDB保存→**フラグメントを即座にhistory.replaceStateで破棄**。成功/失敗をトーストで表示。ファイルピッカー(V-F3)はフォールバックとして残す |

### 1.4 実装上の注意

- 日次JSONは §1.3a で自己定義した契約なので厳格にバリデーションしてよい。一方、export.xml(Apple標準形式)のパーサは**未知のレコード型を無視し、既知の5指標のみ取り込む**寛容設計とする。
- Apple標準XMLの睡眠は `HKCategoryTypeIdentifierSleepAnalysis`、HRVは `HKQuantityTypeIdentifierHeartRateVariabilitySDNN` 等のHK識別子で表現される。識別子→本アプリの指標typeへのマッピング表を1箇所に集約する。
- 睡眠データは複数ソースが混在する(実データで確認: iPhone / Apple Watch / Sleep Cycle / SOXAI RING / Clock の5種)。同一夜に複数レコードがある場合は**ソース別に保持し、表示時は合算ではなく優先ソース選択**(設定で任意ソースの優先順位を選べる)とする。実睡眠時間の計算では `InBed` ステージを除外する。HRVはApple Watchのみが書き込むことを確認済み。
- sourceName文字列にはノーブレークスペースが含まれる場合がある(例: "USER's Apple\u00a0Watch")。ソース名の比較・表示時に正規化すること。日次JSON(§1.3a)ではsourceが空文字列の場合があり、これも許容する。
- 睡眠ステージは今回スコープ外。ただしデータモデルは `sleepStage` フィールドを持てる形にし、拡張余地を残す。
- タイムゾーンはAsia/Tokyo固定でよい。睡眠の日付帰属は**レコード単位ではなく連続セッション単位**で行う: 隣接レコード(前のendと次のstartの間隙が閾値以内)を1セッションに束ね、**セッションの開始時刻**で帰属日を決める(正午前開始→前夜扱い)。実データでセッションが正午をまたぐ例を確認済みのため、レコード単位の正午判定は使わないこと。

### 1.5 受け入れ条件

- 実機iPhoneの既定ブラウザ(Chrome)でホーム画面追加後、オフラインで過去データが閲覧できる
- ショートカット出力の実JSON、および実export.xml(の抜粋)を取り込み、5指標すべてが月表示で描画される
- 同じファイルを2回インポートしても重複データが生じない
- `#import=` 付きURLで開いた際に自動取込が完走し、アドレスバー・履歴にフラグメントが残らない(実機確認)
- Plan B → Plan A の順で取り込んだとき、重複期間のデータが二重計上されない

## 2. 機能③ ジムログ(記録 + 可視化 + プログラム実行支援)

### 2.1 目的

長期目標「筋肥大による体づくりとその維持」に向けて、(a) トレーニングの記録を本アプリ内で行い、(b) 蓄積データから進捗を可視化し、(c) 外部で設計された週間トレーニングプログラムのスケジューラ・通知として機能する。**Gym get-fitからの乗り換え(決定済み)**。過去データの移行は行わず新規スタートとする。

**役割分担(重要)**: トレーニングメニューの設計・食事方針の策定・KPI目標値の設定は本アプリの機能ではなく、月次のClaudeコーチングセッション(人間+LLM)で行う。本アプリはその成果物(§2.3a プログラムJSON)を取り込み、実行を支援する。「コーチの頭脳」は作らず「処方箋のスケジューラ+記録係」に徹する。

### 2.2 記録UXの設計原則(自動化ファーストの適用)

- ジムでの1セット記録は**3タップ以内**を目標とする(種目選択→重量→回数の確定)
- 当日メニュー(G-F6)がある日は、開いた時点で**今日の種目リストが記録画面にプリセット**されており、種目選択が不要
- 重量・回数の初期値は**同種目の前回値**を自動セット(増減はステッパーで±刻み調整)
- 記録はIndexedDBに即時保存。①と同じバックアップ機構(§5)の対象に含める

### 2.3a 週間プログラムJSON(コーチングセッションの成果物。本プロジェクトで定義する契約)

```json
{
  "schemaVersion": 1,
  "programName": "2026-07 筋肥大期 v1",
  "validFrom": "2026-07-06",
  "validUntil": "2026-08-02",
  "weeklySchedule": [
    {"dayOfWeek": "Mon", "focus": "胸・三頭", "exercises": [
      {"name": "ベンチプレス", "sets": 3, "reps": "8-10", "note": "前回e1RMの75%目安"},
      {"name": "インクラインダンベルプレス", "sets": 3, "reps": "10-12", "note": ""}
    ]},
    {"dayOfWeek": "Wed", "focus": "背中・二頭", "exercises": []},
    {"dayOfWeek": "Fri", "focus": "脚・肩", "exercises": []}
  ],
  "nutritionTargets": {"proteinGramsPerDay": 140, "note": "簡潔メニュー方針"},
  "bodyMetricsToTrack": ["weight", "bodyFatPct", "chest", "arm", "waist", "thigh"]
}
```

### 2.3 機能要件

| ID | 要件 | 詳細 |
|---|---|---|
| G-F1 | セット記録 | 種目・重量(kg)・回数を1セット単位で記録。§2.2のUX原則(3タップ以内・前回値プリセット・当日メニュー連動)に従う。記録の編集・削除も可能 |
| G-F1b | 種目マスタ | 種目の追加・編集(名称・部位)。プログラムJSON取込時に未知の種目は自動でマスタに追加 |
| G-F2 | 部位マッピング | 種目→部位(胸/背中/脚/肩/腕/コア)の対応表。初期マッピングを内蔵し、UI上で編集可能 |
| G-F3 | 重量推移グラフ | 部位ごとに: 横軸=時間(日)、縦軸=重量(kg)。**その日の使用重量の中央値をプロットし、最低〜最高重量をエラーバーで表示** |
| G-F3b | e1RM推移グラフ | 主要種目ごとに推定1RMの推移を表示。計算式はEpley(`e1RM = 重量 × (1 + reps/30)`)をデフォルトとし、その日の最大値を採用。**目標管理の主指標** |
| G-F4 | 期間切替 | ①と同じ(週/月/3ヶ月/年)。過去データ全体を含む |
| G-F5 | 種目別ドリルダウン | 部位グラフから種目単位のグラフへ遷移 |
| G-F6 | プログラム取込 | §2.3a のJSONをインポートし、今週のスケジュール・今日のメニューを表示。有効期間(validFrom/Until)で世代管理 |
| G-F7 | 通知(カレンダー連携) | **主方式(購読)**: プログラムJSONのリポジトリコミットを起点に、GitHub Actionsが.icsを自動生成しGitHub Pagesの固定URLに配置。Googleカレンダーで当該URLを購読(初回1回)し、以降の改訂は自動反映。イベントは (a) 週初め: 今週の全体メニュー、(b) 各トレ日の朝: 当日メニュー(標準通知付き)。**副方式(フォールバック)**: アプリ内で.icsを生成しダウンロード提供(手動インポート用)。**Web Pushは使わない**。購読方式はメニューが公開URLに載るため、README等でその旨をユーザーに明示すること |
| G-F8 | ボディ記録 | 週1回想定の手入力: 体重 / 体脂肪率 / 周囲径(胸・腕・ウエスト・大腿)。①と同じ時系列チャートで表示 |
| G-F9 | 栄養の簡易記録 | 日次の「タンパク質目標 達成/未達」の二値タップ記録のみ(グラム入力は不要)。達成率を週次表示 |
| G-F10 | KPIサマリー | 1画面に集約: 直近のe1RM(主要種目)、週あたり部位別セット数、トレ実施率(プログラム対比)、体重・体脂肪率トレンド、タンパク質達成率、睡眠時間(①のデータを参照) |
| G-F11 | コーチング用エクスポート | 月次セッションに持ち込むためのサマリーJSON/テキストを生成(期間内のe1RM変化、部位別ボリューム、実施率、ボディ記録、睡眠要約) |

### 2.4 スコープ外(明示)

- アプリ内でのメニュー自動生成・提案(月次Claudeセッションの担当)
- YouTube等の外部コンテンツ参照機能(同上。方法論の反映はプログラムJSONを通じて行う)
- 食事メニューの提案・カロリー計算(タンパク質二値記録のみ)
- Web Push通知(icsで代替)

### 2.5 受け入れ条件

- 記録画面で1セットの記録が3タップ以内で完了する(実機確認)。当日メニューがある日は種目がプリセットされる
- 同一日に同一種目を複数セット記録した場合、中央値・最小・最大・e1RMが正しく計算され、部位別の中央値+エラーバーのグラフとe1RM推移が描画される
- サンプルのプログラムJSONのコミットからActionsが.icsを生成・配置し、Googleカレンダーの購読で予定と通知が反映される(実機確認)
- KPIサマリー画面に①の睡眠データが反映される(①未取込時は非表示で劣化)

## 3. 共通層(core)への要求

グラフ描画は①③④で共用するため、以下を `src/core/charts/` に共通コンポーネント化する。

- 日次時系列チャート(範囲切替、2軸重ね合わせ、エラーバー対応)
- **エラーバーはライブラリによってはネイティブ対応がない。**Rechartsなら `ErrorBar` コンポーネントで対応可能。ライブラリ選定時にG-F3の実現可否を最初に検証すること。
- インポート基盤(ファイルピッカー → パーサ登録制 → 差分保存)も共通化する。①のJSON/XML・③のプログラムJSON・④の将来データが同じ枠組みに乗ること。

## 4. 機能④ 習慣化

### 4.1 目的

ランニング・ジム・生活習慣・語学学習の習慣化を支援する。設計はdeep research文書に基づく(詳細仕様: `docs/habit-algorithm.md`)。

### 4.2 アーキテクチャ上の絶対条件

- エンジンは `packages/habit-engine/` に**UI非依存の純粋関数**として実装する。四本柱:
  - `canonicalize(rawActivity) -> HabitSpec`
  - `buildLadder(spec) -> PlanLadder`
  - `generatePlan(user, specs, history) -> WeeklyPlan`
  - `applyObservation(state, observation) -> HabitState`
- エンジンには単体テストを必須とする(UIより先にエンジン+テストを完成させる)
- v1はルールベース+軽量確率モデルのみ。**full RLおよびcontextual banditは実装しない**(将来フェーズ)

### 4.3 v1機能要件(UI)

| ID | 要件 | 詳細 |
|---|---|---|
| H-F1 | 活動登録 | 活動名 / 週の目標頻度 / 使える文脈(時間窓・アンカー行動) / 価値理由 を入力 → HabitSpecへ正規化 |
| H-F2 | 分解ラダー | 各活動に最小/標準/伸長/復旧の4実行単位を生成・編集可能 |
| H-F3 | 週間計画 | 制約付きgreedyで週間計画を生成・表示。event-based cue(「朝食後に」等)を時刻より優先 |
| H-F4 | 日次チェックイン | 完了/未完了、努力感(3段階)、文脈一致の記録。1回の操作は10秒以内で完了する導線 |
| H-F5 | 進捗表示 | binary streakではなく**momentumスコア**(減衰付き)を主表示。未実行1回でゼロにしない |
| H-F6 | 失敗回復 | 未実行が続いた活動には復旧単位を提示 → 難度縮小 → cue再設計の順で介入 |
| H-F7 | 週次レビュー | 完了率・文脈一致率・自己開始率のサマリー。文言は評価ではなく観察ベース |

### 4.4 通知について(制約)

- PWAのWeb Push(iOS 16.4+)は**フェーズ3の最後に検証タスクとして実施**。動作が不安定な場合、v1は「アプリを開いた時に今日の計画を表示する」のみとし、通知に依存しない設計とする(habit-algorithm.mdの設計思想とも整合: 通知は初期補助であり、event cueへの移行が本筋)。

### 4.5 受け入れ条件

- habit-engineの単体テストが全通過(コールドスタート時の週間計画生成を含む)
- 「英語学習 週5 / 筋トレ 週3 / 家計記録 毎日」のサンプル入力から一貫した週間計画が生成される
- 14日分の観測を与えたとき、HabitStateの段階遷移(initiate→stabilize等)が仕様通りに動く
